# frontend/src/components/database

> Modals and helpers for creating and managing Snowflake schema-level objects: databases, tables, columns, stages, file formats, insert mapping, and single-row inserts.

## Responsibility

Hosts the wizard-style creation modals and property sheets for schema objects accessible via the sidebar context menus or toolbar actions. SQL is always generated by the Go backend; these components only collect configuration and display SQL previews.

## Files

| File | Purpose |
|---|---|
| `AddColumnModal.tsx` | `ADD COLUMN` wizard; mirrors `column.AddColumnConfig` (name, type, default value, identity, constraints, collation, comment); calls `BuildAddColumnSql` for a live SQL preview and `ExecDDL` to apply; uses `DataTypeSelect` and `ObjectNameCaseControl` from `shared/`. Column DDL is built entirely in `internal/column`. |
| `CreateDatabaseModal.tsx` | `CREATE DATABASE` form with optional clone, transient flag, data retention slider, external volume, catalog integration, and tag assignments; calls `ExecDDL`; uses `ObjectNameCaseControl`. |
| `DatabasePropertiesModal.tsx` | Editable Database properties (opened from the sidebar `db` "Properties" item). Loads `GetObjectProperties(db, "", "DATABASE", name)` (SHOW DATABASES) plus `GetDatabaseParameters(db)` (SHOW PARAMETERS, the source for parameters SHOW DATABASES omits). Renders inline-editable Comment / Data retention / Max data extension / Default DDL collation / Rename; a **Tags** section (shared `TagsRow`, current tags via `GetObjectTagReferences("DATABASE", db, "", name, "")`); **Storage & Iceberg** and **Notebook & Streamlit** live-list `PickerRow`s (External volume, Catalog, Catalog sync, notebook compute pool CPU/GPU, Streamlit warehouse) plus Iceberg params; a **Governance** section (`EVENT_TABLE` `PickerRow` via `ListEventTables` (account-wide `SHOW EVENT TABLES`, fully-qualified names quoted per-part); `CLASSIFICATION_PROFILE` free-text `EditRow` — a `parseString` value, so no list IPC; a `CONTACT <purpose> = <contact_name>` set/unset-by-purpose editor with the purpose spliced unquoted so validated as a bare identifier; a multi-line `DATA_QUALITY_MONITORING_SETTINGS` YAML editor; and an `UNSET DCM PROJECT` button); a **Parameters** section of fixed-choice `SelectRow`s; a **Replication & Failover** section (`ENABLE / DISABLE REPLICATION / FAILOVER TO ACCOUNTS`, `REFRESH`, `PRIMARY`) driven by a validated account-list input (strict `org.account` pattern — the value is spliced unquoted into the clause, so a stray `;`/quote disables the action rather than reaching Snowflake); and a **Danger zone** `SWAP WITH` (sibling-database picker via `ListUserDatabases`, which excludes shared/imported databases + confirm). All applied via `AlterDatabase(db, clause)`. The row components (`EditRow` / `InfoRow` / `SelectRow` / `PickerRow`) and helpers (`q1` / `opts` / `paramValue`) come from `../shared/PropertyRows.tsx`, shared with `SchemaPropertiesModal`; databases are one-level so there is no schema part and no Managed access toggle. **Deferred**: `ADD/REMOVE/MODIFY REPLICA` (needs a bespoke replica editor). |
| `CreateTableModal.tsx` | `CREATE TABLE` form with column definitions, clustering keys, transient/temp flags, and comment; calls `ExecDDL`. The per-column Default field pairs a free-text `Input` with a `DefaultFunctionPicker` (from `shared/`) — a searchable, category-grouped dropdown that inserts a built-in function default (`CURRENT_TIMESTAMP()`, `UUID_STRING()`, …). Function defaults are only valid at table-create time, so this shortcut is not offered in `AddColumnModal`. Pass `onDefine(cfg)` to make the modal **return** its exported `TableConfig` instead of executing DDL — the ER Designer uses this for new-table creation (#615). SQL is built via `shared/tableDdl.ts` (`createTableClause`/`tableOptionsClauses`) and `shared/columnDdl.ts`. |
| `CreateStageModal.tsx` | `CREATE STAGE` form for internal and external stages; configures URL, credentials, file format, copy options, and server-side encryption; calls `ExecDDL`. |
| `CreateFileFormatModal.tsx` | `CREATE FILE FORMAT` builder; delegates format-specific field rendering to `FileFormatFields`; calls `ExecDDL`. |
| `FileFormatFields.tsx` | Renders format-specific fields (CSV, JSON, Parquet, Avro, ORC, XML) as a reusable sub-component used by `CreateFileFormatModal` and `StagePropertiesModal`. |
| `FormatPreviewTable.tsx` | Reads a sample of stage files using `PreviewStageFile` and renders the result in a table; used inside `CreateFileFormatModal` for live previewing. |
| `StagePropertiesModal.tsx` | Displays and edits stage properties (`GetObjectProperties`, `AlterStage`); allows changing URL, integration, file format, and copy options inline. |
| `StageBrowserModal.tsx` | Virtual-scrolled table of stage files (`ListStageFiles`); supports multi-select, `RemoveStageFiles`, and `DownloadFileFromStage`; uses TanStack Table + `react-virtual`. Download/delete are always available (the `getCommand`/`removeCommand` flags were removed in issue #567). |
| `ObjectSummariesModal.tsx` | Ant Design `Table` showing `table.TableSummary[]` for all tables in a database; calls `GetDatabaseTableSummary` (backed by `internal/table`). |
| `InsertMappingModal.tsx` | Visual column mapper for `INSERT INTO … SELECT`; reads column+type info via `GetTableColumnsWithTypes`; produces a SQL snippet inserted into the active editor tab; uses `useInsertMappingStore`, `useObjectStore`, `useQueryStore`. Always available for users; admin policy can still force it off via the `insertMapping` flag. |
| `InsertRowModal.tsx` | Per-column grid form to `INSERT` **one or more rows** into an existing table (distinct from `InsertMappingModal`'s table-to-table `SELECT`). Enumerates columns via `GetTableColumnsWithTypes`; the grid has one row per record and one column per table column, with an **Add row** / delete-row control. Each cell is a literal **Value** (rendered per data type via `InsertCellInput`), a raw **Expr**, **NULL** (nullable columns), or **DEFAULT**, chosen from a per-cell dropdown that also lists the built-in function shortcuts. SQL is built in Go via `BuildInsertRowsSql` (`internal/table`) for a live `SqlPreview` and executed with `ExecDDL`; uses `CreateModalShell` + `useSqlPreview`/`useCreateSubmit` from `shared/`. Always available for users; admin policy can still force it off via the `insertRow` flag. |
| `InsertCellInput.tsx` | The per-cell **Value**-mode widget for `InsertRowModal`, chosen from the column's parsed data type: numeric input (precision/scale hint), TRUE/FALSE select for booleans, AntD `DatePicker`/`TimePicker` for date/time/timestamp (zone-aware format for `TIMESTAMP_TZ`/`_LTZ`), a JSON textarea for `VARIANT`/`OBJECT`/`ARRAY`, an array box for `VECTOR`, hex input for `BINARY`, UUID/WKT inputs for `UUID`/geospatial, and a plain input (with max-length hint) for text. The JSON and geospatial textareas carry a `SnippetToolbar` — a **Template** picker (JSON scaffolds / WKT-GeoJSON templates) plus a JSON **Format** action. Writes a plain string back; the Go builder does all SQL rendering, so this is UX-only and surfaces non-blocking validation warnings. |
| `insertCellTypes.ts` | Pure type-classification, validation, and helper functions backing `InsertCellInput`: `parseColumnType` maps a Snowflake data-type string to a `TypeFamily` and extracts length/precision/scale/vector-dimension; `validateNumeric`/`validateJson`/`validateUuid`/`validateHex`/`validateVector` (dispatched by `validateValue`) provide UX-only, non-blocking validation; `snippetsFor` returns the quick-insert templates for a column type (JSON scaffolds for `VARIANT`/`OBJECT`/`ARRAY`, WKT/GeoJSON for geospatial) and `formatJson` pretty-prints valid JSON. Unit-tested in `insertCellTypes.test.ts`. |

## Patterns & integration

- **IPC**: all calls go through `wailsjs/go/app/App`. SQL building for column DDL uses `BuildAddColumnSql` (and related `Build*ColumnSql` methods) from `internal/column`; execution uses `ExecDDL`. Other object creation calls `ExecDDL` directly with the form-assembled DDL string (for objects without a dedicated builder package).
- **SQL preview pattern**: `AddColumnModal` debounces `BuildAddColumnSql` IPC calls to show a live SQL preview below the form — identical to the dbtproject modal pattern. The preview component from `../shared/SqlPreview.tsx` is used for rendering.
- **Shared components**: `ObjectNameCaseControl` and `DataTypeSelect` from `../shared/` are imported by `AddColumnModal` and `CreateDatabaseModal`.
- **Stores**: `InsertMappingModal` reads `useObjectStore` (for object metadata), `useQueryStore` (to insert the generated SQL into the active tab), and `useInsertMappingStore` (mapping state).
- **Feature flags**: after issue #567 these modals are basic functionality. `StageBrowserModal` download/delete and the `CreateFileFormatModal` sidebar entry are always on (their `getCommand`/`removeCommand`/`fileFormatBuilder` flags were deleted). `InsertMappingModal` and `InsertRowModal` have no user toggle but keep admin-lockable `insertMapping`/`insertRow` flags — the effective value is forced on unless IT policy disables it (`config.ForceAlwaysOnFlags`), so the modals read `featureFlagsStore` only to honor an admin lock.

## Gotchas

- Never construct ADD/DROP/RENAME/ALTER COLUMN SQL inline in these components — always call the `Build*ColumnSql` IPC methods from `internal/column`. This is enforced by design: `AddColumnModal` has no SQL template strings.
- `DataTypeSelect` parses and reconstructs type strings like `NUMBER(38,0)` and `VARCHAR(255)` locally; it does NOT call the backend `IsNumeric` / `NeedsQuotes` predicates for its own rendering — those are called by `AddColumnModal` to decide collation field visibility.
- `StageBrowserModal` uses TanStack Virtual for performance; do not replace with a standard Ant Design `Table` for large stage directories.
- `InsertMappingModal` contains a local inline copy of Snowflake reserved keywords for quoting decisions rather than using `ObjectNameCaseControl`; this is a known inconsistency — prefer `needsQuoting` from `shared/ObjectNameCaseControl.tsx` for new work.
- `DatabasePropertiesModal` quoting follows the authoritative `ParseAlterDatabase` grammar in `internal/sqlgrammar/alter.go`, not whether the value looks like an identifier: `EXTERNAL_VOLUME` / `CATALOG` / `DEFAULT_STREAMLIT_NOTEBOOK_WAREHOUSE` take a bare identifier (double-quoted via `quoteIdent`) — note the Streamlit warehouse **differs from the schema modal**, where it is a string literal. `CATALOG_SYNC`, the notebook compute pools, the Iceberg text params, `BASE_LOCATION_PREFIX`, tag values, `LOG_LEVEL` / `TRACE_LEVEL`, and the `'YES'` / `'NO'` of `REPLICABLE_WITH_FAILOVER_GROUPS` need a string literal (`q1`). `TRUE` / `FALSE` / `PRIVILEGED` / `COMPATIBLE` / `OPTIMIZED` are interpolated raw. Replication/failover account identifiers are `org.account` dotted paths passed **unquoted** (the grammar's `parseIdentPath` accepts the dotted form). When adding a row, check the matching `v.option(...)` for `parseScalar`/`parseIdentPath` vs `parseString`.
